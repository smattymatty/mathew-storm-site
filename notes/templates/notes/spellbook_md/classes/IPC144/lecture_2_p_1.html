Session Objectives

By the end of this session, the student will be able to:

  • List the three elements of the Structure Theorem.
  • Identify the symbols used in flowcharts.
  • Recognize the flowchart symbols and their meaning.
  • Construct the elements of the Structure Theorem using Flowcharts.

Flowcharts Background

The third phase of the Program Development Cycle (PDC) is "Develop Solution into an Algorithm". Key points to remember are:

  • Computers cannot devise a plan or decide to act on their own.
  • Computers can only do what they are told, exactly the way they are told.
  • Computers do what you tell them to do, not what you want them to do.
  • An algorithm is a set of precise instructions which describe the task to be performed and the order in which they are to be carried out.

Flowcharting is a tool that helps develop a solution into an algorithm:

  • A picture is worth a thousand words.
  • It uses only a limited number of symbols to define an algorithm of any complexity.

Flowcharts: Shaping the Program

This is where the program starts to take shape. The Structure Theorem states that it is possible to write any computer program by using only THREE basic control structures:

Core Control Structures
  1. Simple sequence: Represents the computer's ability to execute instructions in a step-by-step, sequential manner.
  2. Selection: Represents the computer's ability to make a decision. By evaluating a condition, alternate execution paths are followed.
  3. Iteration: Represents the computer's ability to repeat a series of instructions.

Recall the hierarchy chart: a module is a small program within a larger program that performs a task (a single task if possible). These modules are flowcharts that are invoked by a parent flowchart. The module is dependent on data found in the parent flowchart, and the parent flowchart requires the results of the module to complete its processing.

The Module Call Box is used to call or invoke another flowchart (module). Any variables or values in the current program that are required by the module are listed in parentheses, each separated by a comma. The new flowchart is executed, and when it completes (hits its Stop Terminator), the current flowchart resumes execution. This represents the modularity of programming.
(Visual: Rectangle with double vertical lines on sides.)
* Procedure being called: getData(a, b)
* Function being called (assigning a return value): tax = calcTax(ttl, rate)

Key Definitions for Modularity
  • Variable: A symbol representing any value (a number, a character, or a string of characters). E.g., x = 5y + 10.
  • Value: A fixed, unchanging number or a single character or a string of characters.
  • Parameter: A variable or value that is passed to the module for use in its processing. A parameter list is a list of multiple variables and/or values separated by commas.
  • Return Value: A value generated by a module that can be assigned to a variable in the parent flowchart.
  • Procedure: A module that accepts parameters. If the variables in the parameter list are altered by the module, the changes may affect the variables in the parent flowchart. A procedure may not have a “return value” in the sense of direct assignment like a function.
  • Function: A module that accepts parameters. If the variables in the parameter list are altered by the module, the changes are not reflected in the variables in the parent flowchart (for parameters passed by value). A function may have a “return value”.

All flowcharting rules apply to modules. Modules are flowcharts composed of Simple Sequence, Decisions, and Iteration.
* The START terminal interrupt now shows the name of the module and its parameters (e.g., myModule(a, varB, x, y)).
* The STOP terminal interrupt shows the module is complete and execution control returns to the invoking flowchart.
* For a function, the STOP terminator may be a RETURN terminator, including a value or variable to be returned (e.g., Return q).

Parameters:
Parameters pass values from a flowchart to a module. The names of parameter variables in the module need not be the same as in the calling flowchart, but they are positionally dependent.
Example:
Caller: myModule(a, b, c, d)
Module Start: myModule(a_local, varB_local, x_local, y_local)
(Here a maps to a_local, b to varB_local, etc.)

To identify variable parameters whose contents can be changed by the module and have that change reflected in the caller (pass-by-reference concept), an asterisk (*) might be used in the module's parameter list (convention can vary):
Module Start: myModule(a, b, *c, *d)
(Visual: Module Start oval: myModule(a, b, *c, *d), leading to module logic, ending in Stop oval.)

Selection in Flowcharting

Selection involves "conditions": a logical expression that is either TRUE or FALSE.
The condition is expressed as some form of a mathematical expression. Examples: A > B, SIN(theta) = COS(theta), recordValue = "end of records".

  • Used to show a change in the path of logic.
  • The condition being tested is written inside the diamond.
  • Has two flow lines exiting it.
  • These flow lines MUST be labelled with the result of the decision (i.e., TRUE or FALSE). There must be one TRUE and one FALSE path.
    (Visual: Diamond shape with "A > B + 5" inside. One exit path labeled "True", another "False".)
  • Used to show a change in the path of logic based on equality with one of several values.
  • Has more than two flow lines exiting it.
  • Used for EQUALITY decisions (each flow line is labelled with the value that variable varC must equal for that path to be taken).
  • You may not place a range of values on the flow lines (e.g., 1 - 6 is generally not standard for pure case symbol, though logic can achieve it).
  • You may not place conditions (e.g., x > 5) on the flow lines of a case symbol.
  • One branch may be labelled Default to represent the path taken when none of the other specified values match.
    (Visual: Elongated hexagon or a series of connected diamonds for "varC" with exit paths labeled "1", "2", "3", "Default". A separate "INVALID" example might show paths like "1-6" or "x>5" crossed out.)
  • Used to collect the flow lines that were created from a decision box and restore the logic of the program back to a single stream.
  • THE PATHS FROM A DECISION MUST REJOIN IN A COLLECTOR.
  • A simple circle is often used. One collector for simple (IF-THEN-ELSE) decisions, one for multiway (CASE) decisions before continuing the main flow.
    (Visual: Circle symbol where multiple flow lines converge into one outgoing flow line.)

Detailed Example: Library Overdue Fines

Problem: Help a local Elementary School's library with overdue fines. Librarians enter student's name, book name, sign-out date, and card colour (W, B, G, O). The program calculates days overdue and the fine ($0.05 per day). A list is produced with student name, book name, days overdue, and fine. A procedure daysBetween(date, n) exists: n becomes days between date and today (negative if date is invalid/future).

Card Colour Due Periods:
* White (W): 14 days
* Blue (B): 7 days
* Green (G): 1 day (next day)
* Orange (O): 28 days

IPO Chart:
* Input: studentName, bookName, signoutDate, colourCode
* Process:
1. Get studentName, bookName, signoutDate, colourCode.
2. Call daysBetween(signoutDate, actualDaysBorrowed).
3. Determine allowedDays based on colourCode.
4. IF actualDaysBorrowed < 0 THEN daysOverdue = 0, fine = 0 (handle invalid date)
5. ELSE daysOverdue = actualDaysBorrowed - allowedDays
6. IF daysOverdue < 0 THEN daysOverdue = 0
7. fine = daysOverdue * 0.05
8. Display studentName, bookName, daysOverdue, fine.
* Output: studentName, bookName, daysOverdue, fine

(Flowchart for this example would be complex, involving input, a multiway decision for colourCode to set allowedDays, calling the daysBetween module, calculations for daysOverdue and fine, and output. It would be a good exercise to draw it based on the IPO.)

Iteration in Flowcharting

Iteration is also referred to as "looping." The goal is to repeat a sequence of steps until a certain condition has been met. It's a special case of the decision structure. The sequence of steps in the loop may consist of simple sequences, decisions, or even other loops.

There are two main types of loops:
* Pre-test Loop
* Post-test Loop

The condition that terminates the loop is tested before the loop is entered. It is therefore possible to not execute any of the statements within the loop at all.
(Visual: Flow line -> Diamond (Conditional Expression) -> (True path) Process (Repeated statement(s)) -> Process (Update Statement) -> Flow line back to before Diamond. Diamond -> (False path) -> Exits loop.)

The condition that terminates the loop is tested after the loop is entered. It is therefore necessary to execute the sequence of statements at least once to reach the decision that terminates the loop.
(Visual: Flow line -> Process (Repeated statement(s)) -> Process (Update Statement) -> Diamond (Conditional Expression) -> (True path) Flow line back to before Process block. Diamond -> (False path) -> Exits loop. The True path flow line cannot have symbols embedded on it.)

Flowcharting Best Practices

Integrity of a Flowchart

In this course, you will represent ALL logic necessary to represent a program – every last tedious step. There are no 'magic' shortcuts. A flowchart is not just a collection of symbols and lines.

As a rule of thumb – lines in a flowchart do NOT cross each other.

Think of a flowchart as a story:
* Symbols (rectangles, parallelograms, etc.) are like letters – individually they have little meaning but are used to make "words."
* Structures (Simple Sequence / Decision / Iteration) are like the "words" of the story. You don’t mix symbols arbitrarily, just as you don't mix letters arbitrarily.
* The flowchart is the complete story you are composing from START to STOP.

When combining structures, they must be nested. Each structure (Simple Sequence, Simple Decision, and Iteration) has ONE entry point and ONE exit point. This rule cannot be broken.

Embedding a loop within a decision:
(Visual: Decision diamond. One path of the decision contains a complete loop structure (condition, body, back to condition) before rejoining the main flow after the decision's collector.)

Embedding a decision within a loop:
(Visual: Loop structure. The body of the loop contains a complete decision structure (diamond, true path, false path, collector) before the loop continues or terminates.)

Think of the nesting of your program as a series of self-contained objects, one inside the other. There is no connection/interaction across layers.
(Visual: An "illegal" flowchart example showing flow lines crossing or improperly entering/exiting structures, for instance, a line from outside a loop jumping into the middle of it, or a decision path jumping out of a loop without proper exit.)
